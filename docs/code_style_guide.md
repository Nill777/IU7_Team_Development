# Руководство по стилю кода
Настоящее руководство определяет стандарты написания кода для проекта **Keystroke Biometrics Keyboard & SDK**. 

Цель документа - обеспечение единообразия кодовой базы, повышение читаемости и автоматизация процесса код-ревью. 

Проект придерживается [официальных конвенций Kotlin](https://kotlinlang.ru/docs/coding-conventions.html) и использует статический анализатор **Detekt**. Мердж в ветку блокируется при наличии хотя бы одного нарушения.

## 1. Базовые правила форматирования
Автоматически проверяются линтером.
*   **Длина строки:** Максимальная длина строки — **120 символов**.
*   **Импорты:** Использование Wildcard-импортов (со звездочкой `import java.util.*`) **запрещено**. Каждый класс должен импортироваться явно. Неиспользуемые импорты должны удаляться.
*   **Пробелы:** Запрещены висячие пробелы на концах строк.
*   **Скобки:** Запрещено использование ненужных скобок, например: `if ((a == b))` -> `if (a == b)`.
*   **Магические числа:** На уровне Detekt активировано правило `MagicNumber`, **запрещено** использование в коде числовых литералов без присвоения им имени (например, `if (status == 42)` вместо `if (status == STATUS_ACTIVE)`), необходимо выносить все коэффициенты формул в `private const val`.

## 2. Именование
*   **Пакеты (Packages):** Всегда в нижнем регистре, без подчеркиваний (например, `com.keystroke.domain`).
*   **Классы и Интерфейсы:** `PascalCase`.
*   **Функции и свойства:** `camelCase` (исключение: `@Composable` функции).
*   **Константы:** `UPPER_SNAKE_CASE` (например, `const val MAX_RETRY_COUNT = 5`).
*   **Ненужные абстракции:** Запрещено создавать абстрактные классы, которые не содержат абстрактных методов (`UnnecessaryAbstractClass`). Используйте обычные классы, интерфейсы или `sealed class`.

### Архитектурный нейминг (Clean Architecture)
Для обеспечения легкой навигации по многомодульному проекту применяются суффиксы:
*   **Use Cases:** Глагол + Сущность + `UseCase` (например, `CalculateZScoreUseCase`).
*   **Репозитории:** `I` + Сущность + `Repository` (интерфейс) и Сущность + `Repository` (реализация).
*   **Модели данных:** 
    *   API слой: Сущность + `Dto` / `Request` / `Response`
    *   БД слой (Room): Сущность + `Entity`
    *   Domain слой: Без суффиксов (например, `BiometricProfile`).

## 3. Jetpack Compose
Поскольку в проекте активно используется декларативный UI, применяются особые правила, закрепленные в исключениях Detekt (`ignoreAnnotated: ['Composable']`).

В Kotlin функции по стандарту должны называться с маленькой буквы (например, `fun calculateSum()`). Однако в Jetpack Compose компоненты принято называть с большой буквы (например, `fun ProfileScreen()`), как если бы это были классы.

Без этой настройки Detekt выдавал бы ошибку на каждую Compose-функцию. Параметр `ignoreAnnotated: ['Composable']` позволяет следовать стандартам Google Compose, не нарушая общие правила нейминга в остальном проекте.

**Именование `@Composable` функций:**
*   Если функция возвращает `Unit` (отрисовывает UI), она обязана называться в `PascalCase` как существительное (например, `@Composable fun BiometricPinPad()`).
*   Если функция возвращает значение (вычисляет состояние), она называется в `camelCase` (например, `@Composable fun rememberValidationState()`).

**Бизнес-логика в UI:** Строго запрещено производить математические вычисления внутри Composable-функций. Вся логика инкапсулируется во `ViewModel` и `UseCases`.

## 4. Ограничения сложности кода
Для предотвращения деградации архитектуры, Detekt настроен на строгий контроль цикломатической и когнитивной сложности. Функция, превышающая лимиты, не пройдет CI/CD пайплайн и потребует декомпозиции.

*   **Max Functions (`TooManyFunctions`):** Максимум **15** функций на один класс или интерфейс. Если функций больше, класс нарушает принцип единственной ответственности и должен быть разбит.
*   **Long Method (`LongMethod`):** Функция не должна превышать **60 строк** кода. Длинные функции сложны для восприятия человеком.
*   **Cyclomatic Complexity (`CyclomaticComplexMethod`):** Максимальное количество независимых путей в функции - **10**. Избегайте вложенных друг в друга `if / else` и `when`. Используйте ранний возврат из функции.
*   **Cognitive Complexity (`CognitiveComplexMethod`):** Порог - **15**. Этот параметр оценивает, насколько тяжело *человеку* прочитать код (штрафует за глубокую вложенность циклов `for` внутри `if`).

## 5. Документирование и Комментарии
*   **KDoc:** Все публичные методы и интерфейсы в модуле `:keystroke-sdk` (которые могут быть использованы сторонними разработчиками) обязаны иметь документацию в формате KDoc.
*   **Объяснение НАМЕРЕНИЙ:** Комментарии внутри кода должны отвечать на вопрос **"Почему это сделано?"**, а не "Что здесь происходит?". Код должен быть самодокументируемым за счет правильного нейминга.
